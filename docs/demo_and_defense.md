# 演示与答辩脚本

## 5–6 分钟演示

1. 25 秒：说明校园信息分散、搜索结果难核实，打开问答页展示主动欢迎语和系统边界；
   在历史会话栏搜索“校训”并内联重命名一条会话。
2. 35 秒：普通用户注册登录并只读浏览/预览知识文档；管理员展示用户、文档维护和纠错审核。
3. 45 秒：上传一篇 Markdown，强调 API 立即返回 202，页面展示解析到索引各阶段。
4. 75 秒：提问知识库内问题，展示逐字流式回答、蓝色 `[Sx]`、折叠来源、原文和官网链接。
5. 50 秒：提问本地不能完整回答的校内时效问题，展示黄色 `[Wx]` 联网来源；点赞后再次
   提问，展示隐藏 QA 优先命中、网页归档转为本地 `[Sx]` 且不暴露 QA 条目。
6. 35 秒：提一个明显校外问题，展示礼貌拒答，并说明系统没有进入 `web_search` 阶段。
7. 35 秒：展示倒赞提交正确答案、管理员审核后生成 `USER_CORRECTION` 普通知识文档。
8. 35 秒：展示 Swagger、50 条评测集和 Hit@5/MRR/阈值报告。
9. 20 秒：总结 MySQL 与 FAISS 职责分离、可恢复索引、人工反馈闭环和 Agent 后续路线。

## 建议演示问题

- “2026年9月全国计算机等级考试什么时候报名？”
- “报名对象是谁？”
- “2026年河海大学读书月的主题和专题展览有哪些？”
- 校内联网补充：“河海大学最新招生计划是多少？”
- 校外范围拒答：“请帮我写一个 Python 快速排序程序。”
- 校内无可靠依据：“请告诉我2035年河海大学研究生入学考试的准确日期。”

## 答辩重点

- 为什么不用纯 LLM：模型知识不可控、校园通知时效强、无法给出可核验来源。
- 为什么混合检索：向量擅长语义，BM25 擅长考试代码、日期、部门名；RRF 避免直接
  比较两种不可比分数，Cross-Encoder 用问题—段落联合编码做最后精排。
- 为什么 MySQL + FAISS：MySQL 只管理事务、权限、任务和历史；课件指定的 FAISS 保存并
  检索知识向量，独立知识工件保存正文/metadata/向量以支持恢复。删除文档无需重新嵌入
  其他资料。
- 为什么后台入库：50 MB 文档解析和 Embedding 慢，上传请求不应阻塞；状态机让失败可见、
  可重试。
- 如何降低幻觉：限定资料、低置信拒答、`[Sx]` 编号、生成后校验、来源卡、时效提示。
- 如何控制联网：只有本地证据不足且主题属于河海大学时才联网；其他问题在本地礼貌拒答，
  免费搜索异常也只安全降级，不影响直接 RAG。
- 如何利用反馈：点赞形成去重隐藏 QA 和可追溯原来源，倒赞经过管理员审核后才成为普通
  知识文档；两条路径都不能绕过引用和完整删除机制。
- 如何管理历史：标题搜索在服务端按当前用户过滤，重命名持久化到 MySQL；查看、改名和
  删除均校验会话所有权，不能借 ID 探测或修改他人会话。
- 为什么暂不做 Agent：先建立可测量的 RAG 基线；Agent 增加延迟、成本和不可控工具调用，
  应只处理基础 RAG 已证明不擅长的多跳任务。

## 常见追问

“索引损坏怎么办？”——独立知识工件保存 chunk、metadata 和向量，启动或 CLI 可原子
重建 FAISS/BM25；MySQL 不承担知识正文存储。

“多人同时问会锁住吗？”——检索读取不可变快照；索引锁不覆盖 LLM 网络请求或 SSE。

“为什么相似问题能更快？”——隐藏 QA 只负责优先定位已确认答案及其原始来源；模型仍按
当前问题重新组织，最终只引用原知识文档或网页归档，不直接回放历史答案。

“为什么校外问题不联网？”——产品主题固定为河海大学，范围门控既减少无关请求和等待，
也防止系统从开放网页扩张成无法评测的通用聊天机器人。

“如何证明效果？”——固定 50 条数据集，报告 Hit@5、MRR@5、引用正确率、拒答率和 P95；
阈值由数据校准而非拍脑袋。

“Key 如何保护？”——只从 `.env` 读，`.gitignore` 和发布前敏感扫描双重检查；已暴露 Key
必须在供应商后台撤销，Git 删除文件不能替代撤销。

最终 Beamer PPT 应从 `project_journal.md`、架构图、实测报告和页面截图取材，不把尚未实测
的目标写成结果。
