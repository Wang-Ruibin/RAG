# 演示与答辩脚本

## 5–6 分钟演示

1. 30 秒：说明校园信息分散、搜索结果难核实，展示目标与安全边界。
2. 40 秒：学生注册登录；管理员登录后展示用户角色和账号启停。
3. 50 秒：上传一篇 Markdown，强调 API 立即返回 202，页面展示解析到索引各阶段。
4. 90 秒：提问知识库内问题，展示逐字流式回答、`[Sx]`、原文片段、日期和官网链接。
5. 40 秒：继续追问，展示会话上下文和历史持久化。
6. 30 秒：提一个知识库外或未来问题，展示低置信拒答。
7. 40 秒：展示 Swagger、50 条评测集和 Hit@5/MRR/阈值报告。
8. 30 秒：总结 MySQL 与 FAISS 知识库职责分离、可恢复索引设计和 Agent 后续路线。

## 建议演示问题

- “2026年9月全国计算机等级考试什么时候报名？”
- “报名对象是谁？”
- “2026年河海大学读书月的主题和专题展览有哪些？”
- 无答案：“请告诉我2035年研究生入学考试的准确日期。”

## 答辩重点

- 为什么不用纯 LLM：模型知识不可控、校园通知时效强、无法给出可核验来源。
- 为什么混合检索：向量擅长语义，BM25 擅长考试代码、日期、部门名；RRF 避免直接
  比较两种不可比分数，Cross-Encoder 用问题—段落联合编码做最后精排。
- 为什么 MySQL + FAISS：MySQL 只管理事务、权限、任务和历史；课件指定的 FAISS 保存并
  检索知识向量，独立知识工件保存正文/metadata/向量以支持恢复。删除文档无需重新嵌入
  其他资料。
- 为什么后台入库：50 MB 文档解析和 Embedding 慢，上传请求不应阻塞；状态机让失败可见、
  可重试。
- 如何降低幻觉：限定资料、低置信拒答、`[Sx]` 编号、生成后校验、来源卡、时效提示。
- 为什么暂不做 Agent：先建立可测量的 RAG 基线；Agent 增加延迟、成本和不可控工具调用，
  应只处理基础 RAG 已证明不擅长的多跳任务。

## 常见追问

“索引损坏怎么办？”——独立知识工件保存 chunk、metadata 和向量，启动或 CLI 可原子
重建 FAISS/BM25；MySQL 不承担知识正文存储。

“多人同时问会锁住吗？”——检索读取不可变快照；索引锁不覆盖 LLM 网络请求或 SSE。

“如何证明效果？”——固定 50 条数据集，报告 Hit@5、MRR@5、引用正确率、拒答率和 P95；
阈值由数据校准而非拍脑袋。

“Key 如何保护？”——只从 `.env` 读，`.gitignore` 和发布前敏感扫描双重检查；已暴露 Key
必须在供应商后台撤销，Git 删除文件不能替代撤销。

最终 Beamer PPT 应从 `project_journal.md`、架构图、实测报告和页面截图取材，不把尚未实测
的目标写成结果。
